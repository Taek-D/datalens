<task type="auto">
  <name>Task 1: GitHub Actions CI — dual-job workflow</name>
  <files>.github/workflows/ci.yml, backend/pytest.ini</files>
  <action>
    Create the `.github/workflows/` directory at the repo root (not inside frontend/ or backend/) and write `ci.yml`.

    **.github/workflows/ci.yml:**
    ```yaml
    name: CI

    on:
      pull_request:
        branches: [main]

    jobs:
      test-frontend:
        runs-on: ubuntu-latest
        defaults:
          run:
            working-directory: frontend

        steps:
          - uses: actions/checkout@v4

          - uses: pnpm/action-setup@v4
            with:
              version: 10

          - uses: actions/setup-node@v4
            with:
              node-version: '20'
              cache: 'pnpm'
              cache-dependency-path: frontend/pnpm-lock.yaml

          - name: Install dependencies
            run: pnpm install --frozen-lockfile

          - name: Lint
            run: pnpm run lint

          - name: Test
            run: pnpm run test

          - name: Build
            run: pnpm run build

      test-backend:
        runs-on: ubuntu-latest
        defaults:
          run:
            working-directory: backend

        steps:
          - uses: actions/checkout@v4

          - uses: actions/setup-python@v5
            with:
              python-version: '3.11'
              cache: 'pip'
              cache-dependency-path: backend/requirements.txt

          - name: Install dependencies
            run: pip install -r requirements.txt

          - name: Test
            run: pytest
    ```

    **Critical ordering constraints:**
    - `pnpm/action-setup@v4` MUST come BEFORE `actions/setup-node@v4` — otherwise pnpm cache does not activate
    - `pnpm run test` calls `vitest run` (non-watch) per package.json — never `vitest` bare which hangs in CI
    - `--frozen-lockfile` prevents silent lockfile mutation during CI
    - `cache-dependency-path: frontend/pnpm-lock.yaml` — path is relative to repo root (not to working-directory)

    **Note on pytest-asyncio:** The conftest.py uses async fixtures. Add `asyncio_mode = "auto"` to backend/pytest.ini so pytest-asyncio runs all async tests automatically:

    Create `backend/pytest.ini`:
    ```ini
    [pytest]
    asyncio_mode = auto
    ```
  </action>

<task type="auto">
  <name>Task 2: Vercel deployment configuration — SPA rewrite + env stub</name>
  <files>frontend/vercel.json, frontend/.env.production, backend/.env.example</files>
  <action>
    Write the Vercel SPA rewrite config and environment variable stubs.

    **frontend/vercel.json:**
    ```json
    {
      "$schema": "https://openapi.vercel.sh/vercel.json",
      "rewrites": [
        {
          "source": "/(.*)",
          "destination": "/index.html"
        }
      ]
    }
    ```
    Place this file inside `frontend/` (not repo root). Vercel's root directory is set to `frontend/`, so it reads `vercel.json` from there.

    Without this file, direct URL access to any route (e.g., `/upload`) returns Vercel's 404 instead of the React app.

    **frontend/.env.production** (update placeholder with instruction comment):
    ```
    # Replace with actual Render URL after deploying the backend
    # Example: VITE_API_URL=https://datalens-api.onrender.com
    VITE_API_URL=https://your-app.onrender.com
    ```

    **backend/.env.example** (document required env vars for Render dashboard):
    ```
    # Copy to .env for local overrides (never commit .env)
    # Set these in Render dashboard for production

    # Comma-separated list of allowed CORS origins
    # Example: ALLOWED_ORIGINS=https://datalens.vercel.app
    ALLOWED_ORIGINS=http://localhost:5173

    # Optional: pin Python version on Render
    # PYTHON_VERSION=3.11.0
    ```

    Verify the rewrite rule is correct by checking the JSON is valid:
    ```bash
    cd frontend && cat vercel.json | python3 -m json.tool
    ```
  </action>

  <how-to-verify>
    **Step 1 — Commit and push current work:**
    ```bash
    git add .github/workflows/ci.yml frontend/vercel.json frontend/.env.production backend/.env.example backend/pytest.ini
    git commit -m "feat(infra): add CI workflow and deployment config"
    git push origin main
    ```

    **Step 2 — Deploy to Vercel (frontend):**
    1. Go to https://vercel.com/new
    2. Import your GitHub repository
    3. Set Root Directory to `frontend`
    4. Framework Preset: Vite (auto-detected)
    5. Add Environment Variable: `VITE_API_URL` = `https://your-render-url.onrender.com` (use placeholder for now — update after Render deploy)
    6. Click Deploy
    7. Note the deployed URL (e.g., `https://datalens-xyz.vercel.app`)

    **Step 3 — Deploy to Render (backend):**
    1. Go to https://dashboard.render.com/new/web
    2. Connect GitHub repository
    3. Set Root Directory: `backend`
    4. Runtime: Python 3.11
    5. Build Command: `pip install -r requirements.txt`
    6. Start Command: `uvicorn main:app --host 0.0.0.0 --port $PORT`
    7. Add Environment Variable: `ALLOWED_ORIGINS` = `https://datalens-xyz.vercel.app` (your actual Vercel URL from Step 2)
    8. Click Create Web Service
    9. Note the deployed URL (e.g., `https://datalens-api.onrender.com`)

    **Step 4 — Update Vercel env var with real Render URL:**
    1. In Vercel dashboard → Settings → Environment Variables
    2. Update `VITE_API_URL` to your actual Render URL
    3. Redeploy (or it takes effect on next push)

    **Step 5 — Verify deployments:**
    ```bash
    # Test Render health endpoint
    curl -s https://your-render-url.onrender.com/health
    # Expected: {"status":"ok"}

    # Test CORS from Vercel origin
    curl -I -H "Origin: https://your-vercel-url.vercel.app" https://your-render-url.onrender.com/health
    # Must see: Access-Control-Allow-Origin: https://your-vercel-url.vercel.app

    # Test Vercel frontend
    curl -s https://your-vercel-url.vercel.app | head -5
    # Must see: <!doctype html>
    ```

    **Step 6 — Test CI:**
    Create a test PR (or push a trivial change to a branch) — verify both `test-frontend` and `test-backend` jobs show green checks.
